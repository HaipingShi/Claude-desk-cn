# 治理优先级工作流设计文档

> **文档编号:** GOV-2026-06-03-P1  
> **适用范围:** `Claude-desk-cn` 代码治理与重构决策  
> **状态:** 已实施  
> **配套工具:** `.spec-workflow/workflows/governance-priority.py`

---

## 1. 问题背景

`patch_claude_zh_cn.py` 是单文件 6,230 行的单体脚本，承载了 130/157 个图节点。随着 Claude Desktop 官方更新频率加快，补丁维护面临以下挑战：

1. **认知负载过高**：修改任何功能都需理解整份脚本，新人难以贡献
2. **测试覆盖为零**：主脚本无单元测试，修改依赖手工验证
3. **版本分支爆炸**：`patch_cowork_model_menu` (6 个分支) 和 `patch_epitaxy_model_menu` (5 个分支) 合计 1,237 行，每个新版本都要追加新分支
4. **高风险代码集中**：网关认证、凭证处理、模型映射等敏感逻辑与 UI 补丁混在一起
5. **隐性技术债务**：零凝聚度的 `sub1-sub5` 杂项社区说明现有拆分是算法强制切分，而非基于领域边界的合理拆分

**核心问题：在众多待办治理项中，如何科学确定优先级，避免"拍脑袋"决策？**

---

## 2. 设计目标

| 目标 | 描述 |
|------|------|
| **数据驱动** | 基于代码知识图谱（凝聚度、风险评分、依赖数）而非主观判断 |
| **多维加权** | 同时考虑内聚度、业务价值、风险、测试成本、可交付性 |
| **自动计算** | 输入图谱数据，自动输出带分数的排序结果 |
| **可扩展** | 新增维度只需修改权重，无需重设计架构 |
| **可解释** | 每个治理项有明确的评分明细，支持人工复核和调整 |

## 3. 优先级评估模型

### 3.1 评估维度定义

每个治理项在 6 个维度上评分（1-10 分），加权求和得到总优先级分数。

| 维度 | 符号 | 权重 | 数据来源 | 评分逻辑 |
|------|------|------|----------|----------|
| **内聚度** | C (Cohesion) | 20% | code-review-graph 社区 `cohesiveness` | `cohesiveness × 75` 映射到 0-10 |
| **业务价值** | B (Business Value) | 25% | 手工评估（专家判断） | 对项目核心目标的贡献度 |
| **风险暴露** | R (Risk Exposure) | 20% | code-review-graph `risk_score` | `risk_score × 12` 映射到 0-10 |
| **测试缺口** | T (Test Gap) | 15% | 图谱中 `test` 节点关联数 | 无测试=10, 部分测试=5, 充分测试=0 |
| **交付成本** | D (Delivery Cost) | 10% | 节点数 + 跨文件依赖 | 节点越多、跨文件越多=成本越高（反向评分） |
| **可交付性** | S (Shippability) | 10% | 是否可增量交付 | 独立模块=10, 深度耦合=1 |

**优先级分数公式：**

```
P = 0.20 × C + 0.25 × B + 0.20 × R + 0.15 × T + 0.10 × D + 0.10 × S
```

> **D 为反向维度**：交付成本越高，D 评分越低（即分数越低表示越难做，优先级应该相对降低）。

基于 P 分数，治理项分为三层：


## 4. 决策规则

### 4.1 何时启动治理工作流

触发条件（满足任一即可）：

1. **版本升级后**：每次 Claude Desktop 更新导致补丁失效时
2. **图谱重建后**：每次运行 `code-review-graph-plus` 重建知识图谱后
3. **季度回顾**：每季度人工审视一次技术债务
4. **PR 阻塞时**：当改动过大难以 review 时

### 4.2 自动排除规则

以下治理项即使分数高，也应暂缓：

- **上游依赖未完成的**：依赖模块的提取未完成时，下游提取不应开始
- **测试基础设施未就绪的**：没有 CI 跑通前，不做大规模重构
- **临近发布的**：发布窗口前 1 周内不启动新的提取工作

### 4.3 人工覆写规则

以下情况允许人工调整优先级：

- **安全漏洞相关**：任何认证/凭证相关的漏洞发现，自动升至 P0
- **用户投诉集中区**：某功能用户反馈密集，可临时提升优先级
- **社区贡献意愿**：有外部贡献者愿意接手某模块，可配合调整顺序

## 7. 权重调参指南

当前权重基于以下假设：

| 假设 | 权重体现 |
|------|----------|
| 业务价值是最重要的导向 | B 权重 25% (最高) |
| 高风险代码应优先被隔离 | R 权重 20% |
| 内聚度高的模块提取回报高 | C 权重 20% |
| 测试缺口大意味着风险高 | T 权重 15% |
| 交付成本不应过度影响决策 | D 权重 10% (反向) |
| 可交付性影响实际落地 | S 权重 10% |

**调参场景：**

- **安全审计期间**：将 R 提升至 30%，B 降至 20%
- **新人 onboarding 期间**：将 C 提升至 30%，降低认知门槛
- **冲刺清理债务**：将 D 反向评分改为正向（成本越高越优先，因为是大块头）
